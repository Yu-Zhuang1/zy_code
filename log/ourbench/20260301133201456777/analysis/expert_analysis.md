## 2) 专家智能体的聚合决策过程是否正确？依据了什么证据？是否充分？

### 2.1 流程层面：调用顺序与覆盖面
- **优点**：
  - 首批并行调用包含 guard + anchor + 两类 update，符合技能框架的“最小可用集合”。
  - 对时点问题与证据缺口（周末导致 proxy 陈旧、SSE 停复牌不可取）在子因子 IR 中都有披露。

- **缺点/关键缺失**：
  1. **未调用校准因子（cross-source close reasonableness calibrate）**
     - 当前 anchor 的证据链偏“行情工具/搜索摘要”，且 guard/overnight 明确存在时点与可交易性不确定；此时按技能说明，校准因子应当用于“输入不匹配/可比性弱时的收缩”。
  2. **未调用验证因子（factor_verification）**
     - guard 给出 `is_tradable_ok=false` 属于“硬缺口”；若最终仍要输出预测价，至少应尝试一次验证/三角测量（例如其他权威来源确认未停牌/是否ST/是否涨跌停约束等）。

### 2.2 逻辑层面：是否正确处理 guard 结果？
- 技能文件明确：**Guard fail -> `NO_FORECAST` with gate reason**。
- 但专家最终仍输出了价格 `\boxed{93.65}`，且没有任何 gate 说明（受用户格式约束也无法说明）。

**结论**：从“该专家技能契约（SSE_Close_T+1）”角度看，专家的决策逻辑与输出要求冲突：
- guard 明确 fail-closed（无法确认可交易性），按技能应停止并返回 `NO_FORECAST`。
- 专家在未补充验证/校准的情况下直接给出点预测，属于**跳过门禁约束**。

> 需要强调：用户强制要求必须输出数字并使用 boxed 格式，确实会逼迫专家“不能拒绝”。但即便如此，专家仍应在内部调用 `factor_verification` 或 `calibrate` 来尽量弥补门禁缺口，然后再给出低置信预测；日志中未体现这一步。

### 2.3 数值聚合是否自洽？
专家只留下了内部汇总，不展示计算细节。仅从结果 `93.65` 反推：
- 以 anchor 93.00 为基准，预测 93.65 相当于 **+0.70%** 的单日收益。
- 而可用 update 信号：
  - 代理隔夜：-0.2%（且 timing_alignment_ok=0，理论上应强收缩到更接近 0）
  - 公司事件：轻度正向（0.18），但未给出可直接映射到百分比的规则
  - microtrend：近5日漂移为正（约 +1.1%/日对数收益的统计量），若被直接使用会推高预测

**潜在问题**：
- 在“proxy 时点不对齐 + 事件不是强盈利催化”的情况下，最终仍给出 +0.7% 上涨，可能主要依赖 microtrend（近期强势延续）。这在统计上并非不可能，但缺少：
  - 对“周末信息空窗”的收缩（calibrate）
  - 对“可交易性/是否停牌/是否涨跌停约束”的确认（verification）
因此该点预测的证据支撑**不充分**，至少应更靠近 anchor（或给出更强的收缩）。

## 4) 对专家最终输出 \boxed{93.65} 的质量评估

- 数值在 anchor 附近（+0.7%）并未出现离谱跳变，符合“一日预测应锚为主”的常识。

### 4.2 主要风险（高）

- **缺少校准**：没有跨源合理性校验与收缩，导致更新信号可能被过度使用（尤其是 microtrend）。
